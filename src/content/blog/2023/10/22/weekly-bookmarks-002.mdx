### CSS Techniques and News

- [“<u>When to use CSS `text-wrap: balance;` vs `text-wrap: pretty;`</u>”](https://blog.stephaniestimac.com/posts/2023/10/css-text-wrap/) by [Stephanie Stimac](https://stephaniestimac.com/) — a comparison of two different values of `text-wrap` currently available in Chrome, alongside their use cases.

- [“<u>CSS Text Level 4 Draft Updated</u>”](https://www.w3.org/blog/CSS/2023/10/20/css-text-level-4-draft-updated/) by [Florian Rivoal](https://florian.rivoal.net/) — an important thing to mention in addition to the previous article, as the `text-wrap` naming would change to `text-wrap-style` eventually. Unfortunate that it did ship with the current name, with people already adopting it, but the new naming is a bit more clear.

- Gecko’s [“<u>Intent to ship: CSS `text-wrap: balance`</u>”](https://groups.google.com/a/mozilla.org/g/dev-platform/c/uAscf5sfEdc/m/0EQcFYqxAAAJ) — related to the previous two articles: it seems that Firefox wants to ship the `text-wrap: balance` anyway, and then later adjust to the new naming, making the situation slightly more weird, but also more interop?

- [“<u>A Couple of New CSS Functions I’d Never Heard Of</u>”](https://chriscoyier.net/2023/10/17/a-couple-of-new-css-functions-id-never-heard-of/) by [Chris Coyier](https://chriscoyier.net/) — a nice little _round_-up of new CSS functions. I actually never heard about `xywh()`, and I already like how its name describes its API. Maybe we could rename `border-image` property accordingly, eh?

- [“<u>@supports @property</u>”](https://codepen.io/propjockey/pen/OJdJmya?editors=1100) by [Jane Ori](http://propjockey.io/) — a CodePen showing how we can detect the support of `@property`. A bit hacky, as we don’t actually have a native way to detect this, it relies on detecting unrelated features that shipped in the exact same browser versions that `@property` did.

    The ability to test for this natively was resolved by CSSWG in the [“Feature detection for descriptors” issue](https://github.com/w3c/csswg-drafts/issues/2463#issuecomment-1016720310), though there are still unresolved issues like [testing for preludes](https://github.com/w3c/csswg-drafts/issues/6966). Browsers are yet to implement this, though, and hopefully when they do, we will be able to test for things like `@property` natively.

- Chromiums’s [“<u>Intent to Ship: CSS Scrollbars: `scrollbar-color`, `scrollbar-width`</u>”](https://groups.google.com/a/chromium.org/g/blink-dev/c/PkEsMirl2zE/m/ArphSa39AAAJ) — I’m not sure yet what I think about these — I understand that it is unlikely we’d get the much more powerful WebKit’s pseudo-elements, so having at least _some_ standard way to control the scrollbars would be nice, _but_ I’d really want to see the `scrollbar-width` accepting `<length>` instead of its current `auto | thin | none`.

    I think one of the reasons not to have it was to prevent the scrollbars being too thin… But what if we’d just say “Treat any length value as something like `max(thin, my-value)`”? Basically, have the value that the scrollbar can get via `thin` be the lower bound, so we could still control the scrollbar width in a way that it would be consistent with other design elements.

- Gecko’s [“<u>Intent to prototype & ship: CSS text-indent keywords `hanging` & `each-line`</u>”](https://groups.google.com/a/mozilla.org/g/dev-platform/c/z6PaiICBCRg/m/w171E3_DAAAJ) — can be very useful for wrapping `<code>` elements (not to be confused with [hanging punctuation](https://drafts.csswg.org/css-text/#hanging-punctuation-property)).

- Chromiums’s [“<u>Intent to Ship: CSS Exponential Functions</u>”](https://groups.google.com/a/chromium.org/g/blink-dev/c/oAu01pBscs8/m/JUaPY3KwAQAJ) — I always welcome more maths into CSS, even though I don’t use it very often. But I know a lot of others do, and sometimes maths can unlock some interesting effects!

- [“<u>Scroll-Driven State Transfer</u>”](https://kizu.dev/scroll-driven-state-transfer/) — My fourth article about scroll-driven animations, where I explore how we can transfer the state of one element to a completely different place on a page by connecting them with a unique identifier in CSS via a timeline-scope.

### Not Exactly New, but Noteworthy

- Some links from [a short Mastodon Thread about multicol](https://front-end.social/@kizu/111258253160862526) by me:

    - [“<u>CSS Multi-Column Layout block element breaking</u>”](https://github.com/web-platform-tests/interop/issues/520) by [Scott Kellum](https://scottkellum.com/) — a “focus area proposal” for Interop 2024. I recommend voting for this issue — any improvements to multicol would be really awesome, it is a highly overlooked area in CSS.
    - [“<u>Things We Can’t (Yet) Do In CSS</u>”](https://www.smashingmagazine.com/2019/11/css-things-cant-yet-do/) by [Rachel Andrew](https://rachelandrew.co.uk/) — a 5 years old, but still actual article about various missing CSS multicol features. Curiously, some of them might be possible with the help of some of the newer CSS specs, though in a rather hacky way. I might write about this relatively soon.
    - [“<u>Overflow in the block direction for continuous media</u>”](https://github.com/w3c/csswg-drafts/issues/2923) also by [Rachel Andrew](https://rachelandrew.co.uk/) — a CSSWG issue about one of these issues which is almost impossible to do without a native support for it. From what I can remember, there are a lot of issues with fragmentation in CSS, and I think there was some work going on to improve it in general? Hopefully, one day, this would be sorted.

- [“<u>CSS 2</u>”](https://drafts.csswg.org/css2/) — if you can’t find something obvious in the specs, try looking in CSS2.

    A lot of things did not move to the new levels of specs, so a lot of basics and their nuances are still only there in CSS2.

- [“<u>CSS Indexes</u>”](https://drafts.csswg.org/indexes/) shared by [Amelia Bellamy-Royds](https://github.com/AmeliaBR/) as [a response](https://front-end.social/@AmeliaBR/111251112262899307) to me sharing the previous tip on mastodon — a listing of every term defined by CSS specs.
